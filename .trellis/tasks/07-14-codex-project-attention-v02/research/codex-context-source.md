# Research — Codex context source
## Sources checked

- Current official Codex manual, Codex App Server section.
- OpenAI `openai/codex` GitHub repository at commit `7790524d6bc79ee3be03ce68941d030684a7b5c0`.
- TypeScript protocol schema generated by local `codex-cli 0.144.1`.
- Live local `codex app-server` request against the user's Codex session history.
- Live ActivityWatch window events on Windows.

## Findings

- The official manual recommends app-server for deep product integrations that need conversation history and streamed events.
- The protocol requires `initialize`, then `initialized`, before `thread/list`.
- Generated `Thread` includes `id`, `name`, `cwd`, `createdAt`, `updatedAt`, `recencyAt`, `status`, and `source`.
- `ThreadListParams` supports source filtering, cwd filtering, `recency_at` sorting, and a state-DB-only fast path.
- Live `thread/list` with `sourceKinds: ["vscode"]` returned root desktop conversations and correctly excluded sub-agent rows.
- The current Windows Codex foreground event is `app=ChatGPT.exe`, `title=ChatGPT`; it contains no project identity.
- Direct SQLite inspection confirms the same metadata exists locally, but it is unnecessary because app-server exposes it officially.

## Decision

Use app-server as the metadata boundary and ActivityWatch as the attention boundary. Persist sparse context transitions locally so the system can reconstruct future switches without inventing past history.
