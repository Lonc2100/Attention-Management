# Attention Management Windows v1.0 连续路线

## Goal

从已验收的 v0.4.2 连续交付 v0.5.0、v0.6.0 与 Windows v1.0.0；每个版本必须是可安装、可回滚、可证明的真实产品增量。

## Requirements

- 每个版本使用独立 Trellis 子任务和版本化自动测试目录。
- 每版遵循：调研 → PRD → 设计 → 红灯测试 → 实现 → 全量门禁 → 真实升级 → 文档 → tag/push。
- 普通产品和技术选择采用保守、可逆默认，不请求用户确认。
- 原始 ActivityWatch 事实不可被展示层或 AI 结论覆盖。
- 所有数据格式变化必须向后兼容并验证真实数据保留。
- Windows v1.0 的外部用户结论若无真实证据，只能交付 RC 和明确验证缺口。

## Child Map

1. `07-15-v0-5-results-and-patterns`：成果与项目关联、证据化复盘、多日规律。
2. v0.6.0：首次启动、隐私控制、备份恢复、导出、诊断与安全升级。
3. v1.0.0：长期稳定、迁移回退、公开文档和发布准备。

## Acceptance Criteria

- [ ] v0.5.0、v0.6.0、v1.0.0 均有独立 PRD、测试证据和版本状态。
- [ ] 每个已完成版本已真实安装运行，历史数据与自启动保留。
- [ ] 所有完成版本的 commit 和 tag 与 `origin/main` 一致。
- [ ] 未经真实验证的能力被标为待验证，不被伪装为完成。
- [ ] 最终产出可由新 AI 仅凭仓库文件继续维护。

## Stop Boundary

证书、费用、真实账号、不可逆数据风险、无法安全处理的 Git 冲突、隐私边界变化、三种修复路径均失败，以及必须依赖真实外部用户或真实 Mac 的验收。
