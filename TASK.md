## 1. 用户任务

从 v0.4.2 起，依次自主完成 v0.5.0、v0.6.0 和 v1.0.0。每版必须完成 PRD、测试、实现、打包、真实升级、数据保留、自启动、文档、Trellis 归档与 GitHub 发布；普通选择不打扰用户。

## 2. 执行边界

- 允许：调研成熟方案、修改代码和文档、安装免费依赖、测试、打包、升级本机应用、commit/tag/push。
- 禁止：删除真实数据、破坏性 Git、付费 API、真实密钥、生产部署、录屏、键盘正文采集、虚构测试和用户验证。
- 决策原则：本地优先、事实层不可覆盖、总量守恒、保守归属、可逆迁移、一个版本解决一个核心问题。

### Batch 6 — 2026-07-17 · v0.7.2 采集可靠性热修复

**完成**：现场取证与对抗性代码审查，建立 v0.7.2 PRD、设计、实施计划和测试门禁；开始 T11。

**关键发现**：

- ActivityWatch v0.13.2 同时向 stdout 和文件写日志，而 Electron 父进程创建 stdout pipe 后未消费；日志累积会把子进程阻塞成“进程/端口存在但 API 不可用”。
- 两次真实日志证明 C 盘写满触发 `SQLITE_FULL`，继而 datastore worker panic 和 poisoned lock；当前 C 盘仍处于低余量。
- 首页 366 日区间查询会周期性制造高负载和大量 flood 警告，既放大 stdout 阻塞，也会在旧服务器全局锁架构下阻塞 watcher。
- v0.7.2 冻结 UI 扩展，优先修复 I/O、增量缓存、数据面健康、磁盘熔断、受控恢复和故障测试。

**实现进展**：

- ActivityWatch stdout 改为忽略；新增进程退出观察、31 日查询上限、历史派生缓存、事件新鲜度和磁盘诊断。
- 自动恢复仅对本应用拥有的 server 生效；三次失败、2 GiB 熔断和 1/5/15 分钟退避均有确定性测试。
- v0.14.0b1 已用独立端口/数据库通过 server、两个 watcher 和查询冒烟；因仍为预发布版，决定继续随包使用 v0.13.2。
- 72 项单元测试通过，3 项需显式外部集成环境的测试按设计跳过；进入构建、E2E、安装升级与 soak 阶段。
- 最终 73 项测试、两轮 17 项 E2E、包内 smoke、真实覆盖安装、数据与自启动保留均通过。
- 15 分钟 0.13.2 已安装链路完成 179 次数据面采样和 60 次 31 日查询；最大桶年龄 140.108 秒，未触发修正后的窗口/AFK 新鲜度门槛。

## 5. 工作假设

- 🟢 H1：成果与项目的显式弱关联比跨应用语义猜测更可信，且不会增加每次切换的操作负担。
- 🟡 H2：至少 3 个已复盘日才适合显示“高质量工作候选时段”；不足时必须明确标注样本不足。
- 🟡 H3：Windows v1.0 的外部可用性无法完全由本机自动化证明，最晚会在真实外部用户门禁处停止。

## 6. 触发用户介入条件

仅使用 Goal Contract 中的 Stop Conditions；普通里程碑不暂停。

## 7. 决策记录

- D1：连续路线采用一个父任务、每个版本一个可独立验收的 Trellis 子任务。
- D2：v0.5 不引入新数据库或分析依赖，复用 ActivityWatch 原始事实和现有分类引擎。
- D3：成果项目关联允许为空；空关联明确显示“无法量化”，不阻止用户完成计划。
- D4：多个成果可以共享项目，因此成果支持时间不做跨成果求和，避免重复时间被伪装成总量。
