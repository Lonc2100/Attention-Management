# 时间效率助手版本规划

> 更新日期：2026-07-17
> 当前可用基线：v0.7.2
> 下一规划版本：v1.0.0
> 产品原则：不只记录“用了什么”，还要判断注意力是否转化为重要成果。

## 1. 产品概述

### 1.1 核心定位

面向个人知识工作者和 AI 工具重度用户的本地优先注意力管理系统。产品自动记录真实电脑活动，识别 Codex 等 AI 工作空间中的具体项目，把时间证据与早间成果计划、实时专注反馈和晚间复盘连接起来。

### 1.2 核心价值

用户不需要手动计时，也不会只得到“Codex 使用了 3 小时”这种粗粒度结论。系统应进一步回答：

- 这段注意力分别流向了哪些项目？
- 哪些时间是可靠归属，哪些仍待确认？
- 连续专注持续了多久，中间发生多少次上下文切换？
- 今天最重要的成果是否真正获得了足够注意力？
- 根据长期规律，明天应该在什么时间推进高强度工作？

### 1.3 初始目标用户

- 第一阶段：产品创建者本人，Windows 单机自用。
- 第二阶段：使用 Codex、编辑器、浏览器和内容工具工作的个人知识工作者。
- 暂不以团队监控、客户计费和雇员工时证明为目标。

### 1.4 与成熟产品的关系

- 采集层优先借用 ActivityWatch，不重复开发窗口与 AFK 采集底座。
- 活动浏览、时间轴、项目颜色和纠错交互借鉴 Timing、ManicTime 等成熟模式。
- 不复刻 Timing 的客户、计费、团队和通用工时产品路线。
- 差异重点是 Codex 项目级上下文识别、成果导向计划复盘、实时注意力反馈和个人规律优化。

## 2. 核心需求清单

| 优先级 | 能力 | 说明 |

| P0 | 可信度表达 | 已确认、推断中、最近确认、待分类不可混淆 |
| P0 | 早间成果与晚间复盘 | 最多 3 个成果、唯一优先项、约 5 分钟复盘 |

| P1 | 注意力驾驶舱 | 统一分母圆环、项目化时间轴、紧凑指标 |
| P1 | 悬浮专注窗 | 当前项目、连续专注、归属状态的实时反馈 |
| P1 | 活动明细与纠错 | 查看完整证据、修正归属、形成规则 |
| P1 | 成果与项目关联 | 判断注意力是否真正支持重要成果 |
| P2 | 个人规律 | 高效时段、切换规律、计划调整建议 |
| P2 | 对外产品化 | 引导、升级、备份、导出、崩溃恢复和签名 |
| P3 | macOS | 产品验证后进入真实 macOS 适配和测试 |

## 3. 版本路线图

### v0.1.0 — 打通真实采集与每日闭环（已完成基线）

**目标**：证明应用不是界面原型，能够真实采集并完成计划到复盘。

**主要能力**：

- ActivityWatch 本地运行时与真实窗口、AFK 采集。
- 早间最多 3 个重要成果和唯一绝对优先项。
- 晚间成果状态、主观效率、总结与明日意向。
- Codex CLI 聚合分析、Windows 开机自启、托盘和补提醒。
- 诊断页和暂停/恢复采集。

**验证点**：安装后可以在真实 Windows 环境连续运行，窗口切换、AFK、计划、复盘和重启恢复均使用真实数据。

### v0.2.0 — 把 Codex 注意力拆到具体项目（已完成基线）

**目标**：解决“Codex 运行了多久，但不知道在做哪个项目”的核心颗粒度问题。

**主要能力**：

- 通过 Codex 上下文识别当前任务、对话与工作目录。
- 将 Codex 前台非 AFK 时间归属到具体项目。
- 项目改名与待分类表达。
- 保留应用事实层，不用项目推断覆盖原始事件。

**验证点**：切换不同 Codex 项目后，时间能够分别累计；离开电脑时不继续计算项目注意力。

### v0.3.0 — 实时看见注意力（已完成）

**目标**：把已有真实数据转化为工作过程中可理解、可持续感知的注意力反馈。

**功能清单**：

1. 今日概览重构为注意力驾驶舱。
2. 以电脑有效活跃时间为统一分母的项目/应用圆环。
3. Codex 按具体项目拆分，同色家族表达父子关系，禁止重复计时。
4. 移除独立 24 小时顶部指针，使用自动聚焦实际活动范围的单轨时间轴。
5. 时间块显示项目、应用证据、准确起止和归属状态。
6. 默认置顶、可折叠、可拖动并记忆位置的悬浮专注窗。
7. 支持置顶、桌面层、临时隐藏和托盘恢复。
8. 严格拆分今日概览、早间计划、晚间复盘、诊断和设置。
9. 补齐加载、空、断连、部分可用、暂停、推断中、最近确认和 AFK 状态。

**不做什么**：

- 不增加活动明细和人工归属纠错。
- 不把其他应用自动强行归入最近 Codex 项目。
- 不增加客户、计费、团队、录屏或手机采集。
- 不开始 macOS 实现。

**验证点**：用户在不打开原始 ActivityWatch 页面、不手动切换项目的情况下，可以从首页和悬浮窗看清当前项目、连续电脑专注、当天项目分布和关键切换时段；断连和低置信度状态不会被伪装成可靠数据。

**对应 PRD**：[v0.3.0 悬浮专注窗与注意力驾驶舱](../.trellis/tasks/07-14-v0-3-attention-dashboard-design/prd.md)

### v0.3.1 — 首屏分布与图表联动优化（已完成）

**目标**：降低首页状态信息占用，让用户打开应用后立即看到全天时间分布，并消除图表颜色和项目名称之间的人工比对。

**主要能力**：

- 压缩当前上下文、记录状态、提醒和摘要指标，在首屏完整展示分布与时间轴。
- 圆环、图例和时间轴使用同一分类键联动高亮。
- 自定义悬浮提示显示项目类型、时长、占比以及时间轴准确起止。
- 支持键盘焦点、窄时间块命中、长名称截断与减少动效偏好。
- 缩小悬浮窗展开态，强化当前项目、连续专注与今日项目占比。

**对应 PRD**：[v0.3.1 首屏分布与图表联动优化](../.trellis/tasks/07-15-v0-3-1-compact-dashboard/prd.md)

### v0.4.0 — 查清并纠正时间归属（已完成）

**目标**：提供类似成熟自动时间追踪产品的调查与纠错能力，但保持项目上下文优先。

**功能清单**：

1. 侧栏新增“活动明细”。
2. 支持按日期查看完整日时间流，而不是只依赖最近事件。
3. 时间线和活动列表联动，直接展示应用、窗口标题、项目与判断来源。
4. 选择时间段后显示准确起止、原始应用、标题、项目归属和归类证据。
5. 支持把待分类或错误时间修正到已有或新建项目，并可撤销。
6. 用户可选择把一次修正保存为后续自动归类规则。
7. 规则使用应用精确匹配与标题包含/完全匹配，支持排序、启停和删除。
8. 人工覆盖始终高于规则；第一条命中规则生效，新规则默认只影响未来。

**不做什么**：

- 不做客户、账单、发票和团队工时。
- 不把截图作为默认证据。
- 不让规则静默覆盖用户的人工修正。

**实现说明**：v0.4.0 不提供正则编辑、批量回溯、项目合并或网站/文档语义层级；这些能力需结合真实使用反馈再决定，不为满足清单而增加不可解释复杂度。

**验证点**：用户可以从一段“待分类”时间追溯到原始证据，完成修正，并验证相同活动下一次能够按规则自动归类；修正不改写原始 ActivityWatch 事实。

### v0.4.1 — 当前打开聊天归属修正（已完成）

**目标**：修正 Codex 项目注意力沿用“最近发起交互任务”的 P0 偏差，让只打开、阅读或输入尚未发送的聊天也能立即获得正确项目边界。

**主要能力**：

- ActivityWatch 继续判断 Codex 前台和非 AFK；Windows UI Automation 读取当前顶部项目名和聊天标题。
- Codex app-server 只负责映射稳定线程 ID 与 cwd，不再把 `recencyAt` 第一项当作当前聊天。
- 切换聊天后在下一次采样内改变归属，无需先发送消息。
- 当前界面缺失、标题无候选或同名无法消歧时进入待分类，绝不继续累计到旧项目。
- 不录屏、不读取键盘正文、不保存界面树，数据格式继续使用 v4。

**验证点**：最近交互任务为 A、当前打开 B 时写入 B；界面不可读时清空 A；完整自动化、打包升级和真实安装链路通过。

### v0.4.2 — 悬浮预览性能优化（已完成）

**目标**：消除首页圆环、图例和时间轴悬浮预览跟随鼠标造成的卡顿，让项目检查稳定、即时。

**主要能力**：

- 提示卡进入目标时一次定位并固定在目标附近，同一目标内移动鼠标不再触发整页状态更新。
- 保留圆环、图例与时间轴的同项目联动高亮，以及鼠标和键盘两种检查方式。
- 移除时间轴批量弱化滤镜和提示卡背景模糊，改用低开销透明度与纯色背景。
- 不改变 ActivityWatch 采集、Codex 项目归属、存储格式和时间计算。

**验证点**：自动化确认提示卡不跟随目标内鼠标移动、联动元素不少于两个、无高开销滤镜，并完成连续两轮 Electron E2E 与真实升级安装。

### v0.5.0 — 让时间对应重要成果（已完成）

**目标**：从“时间记录器”进入“成果导向的注意力管理”。

**功能清单**：

1. 早间成果可关联一个或多个项目，但不要求用户为每次活动手动切换。
2. 首页显示绝对优先成果获得的已确认注意力，而不是只显示总电脑时间。
3. 晚间复盘对照成果状态、项目注意力、连续专注、上下文切换和 AFK。
4. 区分“投入时间很多但没有成果”“时间较少但成果完成”等不同情形。
5. 生成日、周层面的高效时段和项目投入趋势。
6. 建议次日把高强度成果安排在更适合的个人时段。
7. AI 结论必须引用聚合证据并标注数据不足，不能把相关性表述为因果。

**不做什么**：

- 不用单一生产力分数替代成果判断。
- 不因为 AFK 自动判定低效。
- 不以连续坐在电脑前作为高质量工作的唯一标准。

**验证点**：晚间复盘能明确回答“最重要成果是否得到足够注意力、是否完成、明天应调整什么”，而不是只重复应用排行榜。

### v0.6.0 — Windows 对外测试版（已完成）

**目标**：从个人自用工具升级为其他 Windows 用户能够安装、理解、信任和恢复的数据产品。

**功能清单**：

1. 首次启动引导、权限说明和真实采集验证。
2. 可见的隐私说明、排除应用/标题规则和一键暂停。
3. 本地数据备份、恢复与聚合/脱敏导出（已完成；不自动备份 ActivityWatch 原始 bucket）。
4. 明确未签名、无自动更新的安全边界（已完成）；代码签名与公开更新留作 v1.0.0 的证书门禁。
5. 异常退出恢复、数据迁移、日志脱敏和诊断导出。
6. 基础反馈入口、版本说明和可理解的错误文案。
7. 把 Windows 专用能力收敛为平台适配层，核心聚合、项目模型和 UI 不继续写死平台细节。

**不做什么**：

- 不承诺云同步和多人团队。
- 不同时启动 macOS 客户端开发。

**验证点**：新用户在没有开发环境的 Windows 电脑上可以独立完成安装、授权理解、真实采集验证、计划、查看、复盘、备份和升级；卸载与数据处理方式清晰可控。

### v0.7.2 — 采集可靠性热修复（实施中）

**目标**：让 Windows 单机长时间运行时的真实活动采集可诊断、可恢复，并消除全年统计对 ActivityWatch 的周期性压力。

**主要能力**：

- ActivityWatch 子进程 stdout 不再接入无人读取的管道；进程错误与退出会被观察。
- 全年工作活动按最多 31 日分批读取，历史完成日进入可重建缓存，正常刷新只读取当天。
- 诊断同时检查窗口/AFK 桶的新鲜度、磁盘余量和自动恢复状态，不以 `/info` 或端口存活冒充健康。
- 连续三次异常后，仅对本应用拥有的进程执行受控重启；临界磁盘、外部实例和退避期均禁止重启。
- 应用界面与托盘不再等待最长 20 秒的 ActivityWatch 启动。

**不做什么**：不删除或改写真实 ActivityWatch 数据；不在本热修复中改版 UI；不把预发布版 ActivityWatch 直接替换进安装包。

**对应 PRD**：[v0.7.2 采集可靠性热修复](../.trellis/tasks/07-17-v0-7-2-collector-reliability/prd.md)

### v1.0.0 — Windows 正式公开版

**目标**：形成可以长期对外维护的 Windows 正式产品。

**功能清单**：

- 完成 v0.3–v0.6 的稳定整合和真实用户反馈闭环。
- 长时间运行的性能、内存、采集恢复和多显示器稳定性达标。
- 完整帮助文档、隐私政策、第三方许可证和问题反馈流程。
- 版本升级与数据迁移经过回退测试。
- 核心指标、项目归属、历史查询和复盘结论具备自动化回归测试。

**验证点**：产品不依赖开发者现场协助即可被外部 Windows 用户持续使用；升级不会破坏已有历史、计划、复盘、别名或规则。

### v1.x — 产品验证后进入 macOS

**进入条件**：Windows 外部测试已经证明用户持续需要“项目上下文 + 成果闭环”，而不是只需要通用时间统计。

**功能清单**：

1. 为采集运行时、自启动、托盘/菜单栏、路径、Codex 可执行文件和权限建立 macOS 适配。
2. 使用 ActivityWatch 的 macOS 能力，但分别处理 Intel 与 Apple Silicon 运行时。
3. 在真实 Mac 上测试辅助功能、窗口采集、登录启动、睡眠唤醒、多显示器和悬浮窗。
4. 配置 macOS 签名、公证和发布流程。
5. 使用 macOS CI 负责构建，真实设备负责权限与长期行为验收。

**不做什么**：

- 不把“CI 可以生成安装包”误认为“macOS 已经可用”。
- 不在没有真实 Mac 测试的情况下对外发布。
- 不要求第一版同时进入 Mac App Store；发布渠道需结合权限和沙箱约束另行评估。

**验证点**：在真实 Apple Silicon Mac 上完成安装、权限授权、前台采集、AFK、Codex 项目识别、悬浮窗、重启恢复和签名公证验收。

## 4. 关键依赖关系

```mermaid
flowchart LR
  A["真实活动与 AFK"] --> B["Codex 项目识别"]
  B --> C["v0.3 实时驾驶舱与悬浮窗"]
  C --> D["v0.4 活动明细与纠错"]
  D --> E["v0.5 成果关联与个人规律"]
  E --> F["v0.6 Windows 对外测试"]
  F --> G["v1.0 Windows 正式版"]
  G --> H["v1.x macOS 适配"]
```

活动明细依赖完整历史查询和稳定的事实层；成果判断依赖项目归属达到足够可信；对外发布依赖隐私、备份、更新和异常恢复；macOS 只有在产品价值被验证后才值得承担平台适配成本。

## 5. 版本治理原则

1. 每个版本只解决一个清晰问题，并能独立使用和验收。
2. 不为追求功能数量把低置信度结果伪装成自动化成功。
3. 先借用成熟采集和通用交互，再投入差异化的上下文理解与成果闭环。
4. 所有展示层加工都必须满足事实层不可覆盖、总量守恒和可追溯。
5. 涉及原始窗口标题、项目名称和 AI 分析时，默认本地优先并最小化数据暴露。
6. macOS 的“可构建”“可安装”“可长期可靠采集”是三个不同验收层级。

## 6. 下一步行动

1. 在真实日常使用中积累至少 3 个高质量复盘日，检查候选时段是否有解释价值。
2. 进入 v0.6.0：首次启动引导、隐私排除、备份恢复、诊断导出与安全更新说明。
3. 保留代码签名为真实证书门禁；没有证书时明确标注未签名，不虚构可公开无警告发布。
