# v0.4.1 当前 Codex 聊天归属 P0 修正

## Goal

当用户在 Codex 桌面端切换到另一个项目聊天时，时间效率助手应在下一次采样内切换项目归属，而不是继续把注意力累计到最近发送过消息的旧聊天。

## Requirements

- ActivityWatch 继续只负责判断 Codex 是否在前台、用户是否非 AFK；它不承担项目识别。
- Windows 当前可见的 Codex 项目名和聊天标题是项目选择的主信号。
- Codex app-server 的线程列表只用于把当前可见标题映射到稳定线程 ID、cwd 和项目身份，`recencyAt` 不再直接代表当前聊天。
- 切换聊天后，无须发送消息即可更新项目归属，目标响应时间不超过当前 5 秒采样周期加一次检测开销。
- 当前界面不可读取、候选为零或候选不唯一时，不得沿用上一项目；该区间保持 Codex 待分类，并展示可理解的未确认原因。
- 不录屏、不读取聊天正文、不保存窗口树；只在内存中使用顶部项目标签和聊天标题完成匹配。
- 仅支持 Windows；不引入 macOS 兼容承诺。
- 保持 v4 用户数据格式，不迁移、不清空既有记录、纠错和规则。

## Acceptance Criteria

- [x] 当前可见聊天为 A 时，采样写入 A 的线程与项目。
- [x] app-server 的最近交互仍为 A、当前可见聊天切到 B 时，下一次采样写入 B。
- [x] 当前聊天标题相同但项目标签不同，使用项目标签/cwd 文件夹名消歧。
- [x] 当前界面信号缺失或仍有多个候选时，不写入旧项目样本，状态为待分类。
- [x] 非 Codex 前台、AFK 或数据过期时，不调用界面检测，也不写入项目样本。
- [x] UI Automation/PowerShell 失败时应用不崩溃，错误可恢复，后续采样能重新识别。
- [x] 单测、类型检查、构建和 Electron E2E 连续两轮通过。
- [x] v0.4.1 打包安装成功，既有用户数据保留，登录 Windows 后自启动仍可用。

## Notes

- 本修正优先保证“宁可待分类，不要误归属”。
